# Tool Integration Pattern

## Суть
**«LLM думает, API делает».** Внешний API вызывается как детерминированный
инструмент; LLM только анализирует результат. Снижение токенов, повышение
надёжности — там, где у LLM нет эффекта, она не должна работать.

## Проблема
LLM используется для всего — и для мышления, и для рутины: получить посты из
Telegram, проверить статус GitHub PR, прочитать веб-страницу, проверить uptime
сервера. Но:

- LLM **не может** вызвать произвольный API без обёртки (нужны credentials,
  сессии, MTProto и т.п.).
- LLM **галлюцинирует** статусы: «PR смёржен» — а он висит уже неделю.
- LLM **дорогая** для рутины: тратить токены на чтение списка постов —
  перерасход там, где один HTTP-запрос делает то же бесплатно.
- webfetch спасает для простых страниц, но это всё равно прокси через модель,
  а не детерминированный доступ.

## Решение
Агент вызывает **инструмент** — скрипт в `tools/` волта, который через внешний
API получает данные и возвращает результат. LLM только анализирует. Это
[[02-Methods/model-routing|model-routing]] на уровне **источников данных**:
если есть детерминированный способ получить данные — он перекрывает вызов LLM.

### Таблица «LLM vs API»

| Задача | LLM | API |
|--------|-----|-----|
| Классифицировать пост | ✅ думает | — |
| Получить посты из Telegram | ❌ не может | ✅ Telegram API |
| Проверить статус GitHub PR | ❌ галлюцинирует | ✅ GitHub API |
| Прочитать веб-страницу | 🟡 через webfetch | ✅ напрямую |
| Проверить uptime сервера | ❌ | ✅ HTTP request |

Принцип разделения: **мышление** остаётся за LLM, **доступ к данным** уходит к
детерминированному инструменту, который LLM вызывает через bash.

## Реализация в VibeOS
Директория `tools/` в корне волта содержит скрипты-инструменты. Каждый
инструмент — мост к одному внешнему API: получает данные, возвращает JSON для
анализа агентом. Агент вызывает инструмент через bash, без MCP-серверов и
плагинов OpenCode — просто Python-скрипт.

Первый инструмент: `tools/telegram-capture/` (T-062, в разработке) — извлечение
постов из группы @inbox_tools по теме, маркировка реакциями.

## Когда применять
- Есть детерминированный способ получить данные через API/скрипт.
- LLM в этой задаче либо не справляется (не может вызвать), либо галлюцинирует.
- Объём данных велик — прокачивать через токены LLM невыгодно.
- Источник стабилен и документирован (Telegram Bot API, GitHub API, HTTP).

## Когда НЕ применять
- Данных нет в API — нужно творчество LLM (синтез, гипотеза).
- API дорогой/нестабильный — overhead обёртки дороже выигрыша.
- Задача одноразовая и мелкая — дешевле напрямую через webfetch/ask.
- Нужна семантика, которой API не несёт (LLM должна интерпретировать текст).

## Статус внедрения
- **vault** 🟡 — директория `tools/` заведена, первый инструмент
  `tools/telegram-capture/` (T-062) в разработке. Команда `/capture` будет
  извлекать посты → librarian классифицирует.
- **SERPlux** ➖ — коллектор уже детерминированный (Topvisor API), метод не
  формализован как часть `tools/`.
- **dv-hub** ➖ — контекст не предполагает.
- **dotfiles** ➖ — sysop ходит по системе read-only, скрипты-инструменты вне
  фокуса.
