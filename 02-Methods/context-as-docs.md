# Context as Docs (Spec-Driven)

## Проблема
Агент не знает намерений проекта: переизобретает архитектуру, нарушает решения, «галлюцинирует» под отсутствие критериев. Знание живёт в твоей голове, а не в контексте.

## Решение
Документация как инфраструктура (arxiv 2602.20478: «load-bearing artifacts»). Спецификации — не пассивные доки, а исполняемые контракты, которые ограничивают что агент генерит (spec-driven development, ставший стандартом 2026). Папка markdown = «цифровой мозг» проекта.

Два типа контекста (Fowler 2026): **guidance** (конвенции, всегда грузятся — AGENTS.md) и **instructions** (команды под задачу — slash-команды/skills). Не смешивать.

## Реализация в OpenCode
- `AGENTS.md` — guidance: конвенции, стек, запреты. Грузится всегда. Держать КОРОТКИМ.
- `docs/*.md` — архитектура, решения (ADR), глоссарий. Агент читает по ссылке.
- Спека задачи с DoD → отсюда verifier берёт критерии.

Реальные примеры:
- dv-hub: `docs/architecture.md` (ADR), `context/` submodule (vision/kanban) — образцовая реализация.
- SERPlux: `docs/contracts.md` (контракты модулей), `docs/decisions.md`.

## Когда применять / когда НЕ применять
- Применять всегда. Это фундамент, на котором стоят остальные методы.
- НЕ раздувать AGENTS.md: «строить контекст постепенно, не пихать всё сразу» (Fowler). Модели поумнели — половина старых инструкций уже не нужна. Большой контекст = деньги + падение качества.
