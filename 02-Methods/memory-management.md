# Memory Management

## Проблема
Контекстное окно — это RAM, а не диск (Mem0 2026). Главный провал агентов: относятся к окну как к хранилищу. В долгой сессии контекст переполняется, происходит компакция, и важные факты/решения теряются в сжатии.

## Решение
Разделять рабочую память (контекст сессии) и долговременную (файлы на диске). Ключевой приём 2026 — **pre-compaction memory flush**: до того как история сожмётся, агент сбрасывает ключевые факты в файл (MEMORY.md / docs / context). После компакции знание не потеряно — оно на диске.

## Реализация в OpenCode
- Плагин `compaction.ts` (уже есть в dv-hub) — управляет сжатием.
- Перед компакцией важные выводы дописывать в `docs/decisions.md` или `context/`.
- Аналог `/dream` из MiMo: команда, которая по концу сессии сохраняет выводы в docs.

Принцип: окно держать как можно меньшим (Fowler). Не дампить всё — грузить по необходимости через skills/ссылки.

## Pre-compaction flush протокол

Формализация приёма «сброс контекста на диск до того, как окно сожмётся». Это превентивная мера: агент сам инициирует flush до compact, а не ждёт потери.

### Когда срабатывает
- Перед запсанной/плановой компакцией (когда агент чувствует, что окно близко к лимиту, или по явной команде `/flush` / `/dream`).
- В конце длинной сессии перед закрытием.
- Перед передачей контекста другому агенту (handoff) — чтобы следующий поднял с диска.
- НЕ срабатывает на каждый шаг — дорого по токенам. Только в контрольных точках.

### Что делает
1. Агент читает свой активный контекст (ход сессии, принятые решения, diffs).
2. Читает целевой накопитель на диске — что уже записано (избежать дублей).
3. Выделяет **значимое**: ADR, факты, разрешения `[проверить]`, статус задач, блокеры.
4. Дописывает (append-only) в накопитель, не перезаписывая.
5. Сбрасывает рабочее окно: ключевые ссылки на дисковые файлы остаются, дампы — на диск.

### Куда пишет (по проектам)
| Проект | Накопитель | Тригер | Реализация |
|--------|-----------|--------|------------|
| **vault** | `04-Memory/session-log/YYYY-MM-DD.md` + `active-context.md` | инструкция librarian перед compact | `04-Memory/` структура, session-flush плагин |
| **dotfiles** | `.opencode/memory/decisions.md` | команда `/flush` (planner) | новая команда, ADR-формат |
| **SERPlux** | `docs/decisions.md` + `docs/progress.md` | команда `/dream` (build) | compaction.js flush-хук |
| **dv-hub** | `docs/decisions.md` | compaction.ts плагин (план) | flush-протокол ❌ → 🟡 |

### Принципы реализации
- **Append-only** — никогда не перезаписывать существующие записи.
- **Дедупlication** — читать накопитель перед записью, не дублировать.
- **Краткость** — на диск значимое, не весь дамп окна.
- **Не коммит внутри flush** — коммит отдельной командой после verifier PASS.
- **Явный триггер лучше неявного** — команда `/flush` надёжнее авто-детекта переполнения.

### Пример: vault librarian
Перед compact:
1. Прочитать `active-context.md` и последний `session-log/`.
2. Дописать в `session-log/YYYY-MM-DD.md`: что сделано, статус задач, новые факты.
3. Обновить `active-context.md` — убрать выполненное, добавить следующее.
4. Разрешённые `[проверить]` → `facts.md`.
5. После compact — поднять с диска, не из сжатого окна.

### Пример: dotfiles /flush
Planner читает ход сессии + `.opencode/memory/decisions.md`, выделяет ADR и согласованные правки, дописывает выше секции «Compaction flush».

## Когда применять / когда НЕ применять
- Применять: длинные сессии, многошаговые задачи, всё что переживёт компакцию.
- НЕ полагаться на окно для хранения решений — только файлы переживают сессию.
- НЕ flush на каждом шаге — только в контрольных точках (выше).
