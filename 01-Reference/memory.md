# OpenCode Vault: Система памяти

> Вместо claude-mem. OKF-подбандл `04-Memory/` внутри Knowledge Bundle волта.
> Проверено: 2026-06-27.

## Формат: OKF Sub-bundle

Память организована как **OKF v0.1 подбандл** (см. `/Architecture.md` и [index.md](index.md)):
- `04-Memory/index.md` — directory listing подбандла
- `04-Memory/log.md` — хронология изменений памяти
- `04-Memory/active-context.md` — концепт (type: Active Context)
- `04-Memory/facts.md` — концепт (type: Fact Registry)
- `04-Memory/session-log/YYYY-MM-DD.md` — концепты (type: Session Log)

## Зачем

Каждая сессия OpenCode-агента начинается с чистого контекста. Без механизма памяти агент каждый раз:
- перечитывает все файлы волта, чтобы понять контекст
- теряет знания, добытые в предыдущей сессии
- не может отличить новое от уже известного

## Три уровня памяти

### 1. Активный контекст (active-context.md)
Что сейчас в фокусе. Читается librarian при старте каждой сессии.
- какой проект активен
- какие изменения не закончены
- какие `[проверить]` требуют внимания

### 2. Факты и решения (facts.md)
Реестр проверенных фактов. Разрешение `[проверить]` → перемещение сюда.
- факты об OpenCode, подтверждённые документацией
- архитектурные решения по проектам
- ссылки на соответствующие файлы волта

### 3. Сессионный лог (session-log/YYYY-MM-DD.md)
Краткое резюме каждой рабочей сессии:
- что сделано (какие файлы изменены/созданы)
- какие решения приняты
- что осталось на следующий раз

## Протокол работы для librarian

```
Старт сессии:
  1. Прочитать 04-Memory/active-context.md
  2. Прочитать последний session-log (если есть)

В ходе сессии:
  3. При разрешении [проверить] — записать факт в facts.md
  4. При смене активного проекта — обновить active-context.md

Конец сессии:
  5. Создать/дополнить session-log/YYYY-MM-DD.md
  6. Обновить active-context.md (очистить или перевести в done)
```

## Правила
- Файлы памяти — часть волта, версионируются в git.
- Никаких внешних сервисов (claude-mem, mem0, etc.).
- Максимальный размер active-context.md: ~50 строк. Факты — не дублируют Reference, а фиксируют решения.
- Сессионные логи можно архивировать раз в месяц (перенос в `04-Memory/archive/`).
